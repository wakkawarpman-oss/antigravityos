# HANNA NEVERENDING STORY — Майстер-план стабілізації, розгортання та масштабування

## 1. Вступ і рамка рішення

Цей документ є операційним майстер-планом для системи HANNA у режимі стабілізації після інтенсивного циклу змін. Його мета не в тому, щоб вигадати нову архітектуру, а в тому, щоб керовано перевести наявну систему у передбачуваний, повторюваний і безпечний стан продакшн-експлуатації. Ключовий принцип: мінімум ризику, максимум контролю, без хаотичних розширень scope.

План побудований на вже зафіксованих фактах: у проекті є змішаний стек Node і Python, є строгі prelaunch-гейти, є модулі збору даних і артефактні експортери, є окремий OPSEC-контур, і є залежності різної зрілості, включно з підмодулями, що мають підтверджені вразливості. Отже, головний фокус зміщується з «швидко додати ще функцій» на «гарантовано не зламати те, що вже працює, і підняти надійність життєвого циклу релізу».

Аргумент №1: без формалізованої дисципліни релізу будь-яка нова функціональність стає мультиплікатором інцидентів. Аргумент №2: система, яка формує доказові артефакти, не може дозволити собі нестабільні або неперевірені шляхи експорту й валідації. Аргумент №3: у змішаному стеку саме границі між компонентами та залежностями є головною точкою відмови, а не окремий рядок коду.

Тому цей майстер-план націлений на чотири результати: стабільна збірка, чистий шлях розгортання, керована безпека залежностей, і передбачувана експлуатація.

## 2. Цілі та критерії успіху

### 2.1 Стратегічні цілі

1. Досягти повторюваної доставки релізу з однаковим результатом у локальному, CI та продакшн середовищах.
2. Знизити ризик регресій у критичних гейтах (prelaunch, експорт артефактів, STIX-валідація, auth-контури).
3. Відокремити ядро системи від нестабільних або ризикових допоміжних залежностей.
4. Побудувати чіткий runbook аварійного відновлення та операційного контролю.

### 2.2 Вимірювані критерії

1. Три послідовні релізні прогони без ручних «гарячих» правок у pipeline.
2. Нуль блокуючих вразливостей у core dependency lane.
3. Стабільний prelaunch strict verdict без флапінгу між прогонами.
4. Повний комплект артефактів релізу: runtime summary, metadata, JSON/STIX/ZIP, gate verdict, audit snapshot.
5. Mean Time To Detect для помилок гейта — до 10 хвилин, Mean Time To Recovery — до 1 години для проблем конфігурації.

## 3. Базова архітектурна позиція для подальшого розгортання

Надалі система має працювати з розділенням на три операційні площини:

1. Core plane: оркестрація, контракти, API-контури, ключові експортери, release gate логіка.
2. Tool plane: зовнішні модулі збору, адаптери, vendor/third-party інтеграції, інструменти з нерівномірною якістю підтримки.
3. Control plane: preflight/prelaunch, безпекові перевірки, аудит залежностей, моніторинг, журнал рішень.

Аргумент: таке розділення прибирає змішання відповідальності. Якщо падає tool plane, core plane не повинен втрачати цілісність даних і контрактів. Якщо control plane показує fail, це має бути однозначна заборона на реліз, а не «рекомендація».

## 4. Детальний план реалізації по фазах

### 4.0 Операційна політика інструментів (оновлено)

Майстер-план працює у режимі **free-first + freemium-allowed**:

1. Базовий релізний verdict має досягатися без платних ключів.
2. Freemium-інструменти дозволені як додатковий enrichment lane.
3. Вичерпання квот freemium не блокує реліз, якщо baseline-профіль пройшов.
4. Будь-який freemium-модуль повинен мати fallback на безкоштовний контур.

1. Єдиний контракт результату (`RunResult`/normalized hit DTO).
2. Єдиний маршрут у control plane: `preflight -> execution -> verification -> export`.
3. Єдина політика OPSEC: проксі-контур, лог-санітизація, fail-classification.

### Фаза A. Freeze і інвентаризація стану

Мета: зафіксувати точку відліку без шуму.

Кроки:
1. Зафіксувати commit baseline і список релевантних файлів релізу.
2. Зафіксувати dependency snapshot: `requirements.txt`, `package-lock.json`, інвентар tool manifests.
3. Підтвердити актуальність strict gate конфігурації в CI.
4. Зберегти prelaunch bundle як еталон.

Результат: всі подальші відхилення вимірюються від однієї стабільної точки.

Ризик без цієї фази: неможливо відрізнити новий дефект від історичного «дрифтового» стану.

Додатково для інструментальної матриці:

1. Зафіксувати, які елементи йдуть у baseline (free), а які в optional freemium lane.
2. Зафіксувати для кожного елемента відповідний сервіс оркестрації і fallback.

### Фаза B. Dependency hardening і lane-separation

Мета: зробити залежності керованими.

Кроки:
1. Розділити перевірки в CI на core-security lane і tool-security lane.
2. Core lane зробити блокуючим для merge/release.
3. Tool lane зробити обов’язковим для звітності з фіксованим SLA на remediation.
4. Визначити policy версій: для core критичних пакетів — мінімальні патч-рівні з підтвердженим тест-покриттям.
5. Встановити граф ownership: хто відповідає за кожен manifest.

Аргументація:
- Без lane-separation команда або блокує себе через нерелевантні tool-CVE, або пропускає критичні CVE у ядрі.
- Розділення дозволяє тримати швидкість core-розробки і водночас не ігнорувати ризики периферії.

### Фаза C. Релізна цілісність артефактів

Мета: гарантувати, що релізний комплект валідний, повний і відтворюваний.

Кроки:
1. Формально закріпити порядок виконання гейтів: генерація артефактів -> валідація артефактів -> фінальний verdict.
2. Зробити зовнішню STIX-валідацію обов’язковою лише після наявності metadata і export-path.
3. Забезпечити жорстке форматування timestamp і ключових полів STIX для зовнішніх валідаторів.
4. Додати smoke-перевірку для читання/перевірки ZIP маніфесту.

Аргументація:
- Артефактний pipeline без суворого порядку породжує псевдо-fail і псевдо-pass.
- Для доказових сценаріїв найгірша помилка — «звіт є, але юридично або технічно невалідний».

### Фаза D. Надійність розгортання

Мета: зробити deployment рутиною, а не інцидентом.

Кроки:
1. Впровадити deployment профілі: canary, standard, emergency rollback.
2. Для кожного профілю визначити entry-criteria і exit-criteria.
3. Встановити «стоп-прапорці»: якщо gate fail або health-check fail, деплой автоматично блокується.
4. Визначити rollback-procedure без ручного редагування конфігів у проді.

Аргументація:
- Розгортання має бути сценарієм, а не творчою діяльністю on-call інженера.
- Чіткі профілі зменшують когнітивне навантаження і кількість помилок під тиском.

### Фаза E. Операційна стабільність і моніторинг

Мета: мати раннє виявлення відхилень.

Кроки:
1. Уніфікувати ключові метрики: gate pass rate, export success rate, queue latency, worker failure class.
2. Додати розрізнення помилок: конфігурація, залежність, рантайм, зовнішній сервіс.
3. Налаштувати алерти з рівнями важливості та runbook-посиланням.
4. Впровадити щотижневий стабілізаційний review з коротким changelog ризиків.

Аргументація:
- Без розумного моніторингу команда бачить лише «симптом», але не причину.
- Стабільність — це не відсутність помилок, а швидка локалізація і контрольований recovery.

### Фаза F. Оркестрація інструментального контуру

Мета: підключити кожний елемент з матриці через явний сервіс оркестрації і стандартизований interaction contract.

Кроки:

1. Визначити owner-сервіс для кожного елемента (див. матрицю 4.1).
2. Для кожного сервісу зафіксувати `input contract`, `output contract`, `error class map`.
3. Уніфікувати event payload і redaction policy між native, bridge та distributed lanes.
4. Додати acceptance smoke для кожного нового оркестратора: happy path + degraded path.

Аргументація:

- Без явного оркестратора інтеграції швидко перетворюються на неузгоджений набір wrapper-скриптів.
- Явний interaction contract знижує change coupling і вартість дебагу.

## 5. Деталізація подальшого розгортання

### 5.1 Середовища

1. Dev: швидкі локальні перевірки, не джерело фінального verdict.
2. Integration: обов’язкові контракти і ключові regression-тести.
3. Staging: повний prelaunch strict, повний комплект артефактів.
4. Production: деплой тільки з підтвердженим staging verdict.

Правило: заборонено пропускати staging strict prelaunch під приводом «ми і так впевнені».

### 5.2 Порядок деплою

1. Підготовка релізного кандидата (RC): фіксація тегу, dependency snapshot, changelog.
2. Прогін integration + staging strict prelaunch.
3. Перевірка gate-result і final-summary людиною, відповідальною за реліз.
4. Canary rollout на обмежений відсоток трафіку/навантаження.
5. Аналіз метрик канарки (помилки, latency, стабільність воркерів).
6. Повний rollout або rollback за формальними критеріями.

### 5.3 Критерії переходу між етапами

Перехід до canary дозволений лише якщо:
1. Немає required_check_failures.
2. Артефакти повні й валідні.
3. Security lane core не має блокуючих CVE.
4. Стан міграцій/схем даних підтверджено.

Перехід до full rollout дозволений лише якщо:
1. Canary без деградації SLA.
2. Немає нових high-severity помилок у логах.
3. On-call підтвердив готовність rollback шляху.

### 5.4 Rollback стратегія

1. Rollback має бути автоматизованим скриптом, не набором ad-hoc команд.
2. Повернення включає не лише код, а й сумісну конфігурацію.
3. Після rollback обов’язковий post-incident пакет: причина, корінь, запобігання.

Аргументація:
- Найчастіше падає не «код загалом», а різниця між кодом і конфігом.
- rollback без відновлення конфіг-паритету дає лише тимчасовий ефект.

## 6. Операційний контроль якості

### 6.1 Контрольний список перед релізом

1. Git tree чистий або явно задокументований набір змін.
2. Усі потрібні тести зелені, без ігнорованих fail.
3. Prelaunch strict pass у цільовому профілі.
4. Security core lane pass.
5. Tool lane звіт з позначенням відкритих ризиків і строків фіксу.
6. Артефакти релізу завантажені в сховище й доступні для перевірки.
7. Rollback перевірений на останньому стабільному релізі.

### 6.2 Контрольний список після релізу

1. Health-check усіх критичних маршрутів.
2. Моніторинг помилок і latency у перші 30/60/120 хв.
3. Підтвердження коректності обробки черг і воркерів.
4. Перевірка цілісності export-пайплайну на живих даних.
5. Закриття релізу коротким технічним звітом.

## 7. Аргументи щодо управління ризиком і вартістю

### 7.1 Чому не робити великий перепис зараз

1. Великий перепис збільшує інтервал невизначеності й знижує прогнозованість релізів.
2. В поточному стані відомі проблеми переважно операційні: залежності, порядок гейтів, валідність артефактів.
3. ROI від цільових стабілізаційних змін вищий, бо вони одразу знижують частоту інцидентів.

### 7.2 Чому потрібне жорстке правило «один ризик — одна фаза»

1. Воно відсікає змішування причин і ускладнення root cause analysis.
2. Дозволяє виміряти ефект кожної зміни.
3. Зменшує ймовірність каскадних регресій.

### 7.3 Чому dependency governance — центральний елемент

1. Вразливість у транзитивній залежності може повністю знецінити формально «чистий» код.
2. Без ownership manifest-файлів проблема повертається циклічно.
3. Інструментальні модулі можуть бути не в core runtime, але створюють ризик supply-chain і операційної плутанини.

## 8. План робіт на 90 днів

### Дні 1–14

1. Закрити критичні core-залежності, які мають confirmed CVE.
2. Формалізувати lane-separation у CI.
3. Закріпити prelaunch strict policy як обов’язкову.
4. Оновити релізний runbook.

### Дні 15–30

1. Пакетно оновити tool manifests з найвищим ризиком.
2. Додати smoke-сценарії для інструментів після оновлень.
3. Вирівняти шаблони логування й класифікацію помилок.

### Дні 31–60

1. Впровадити canary discipline і SLA-based auto-stop.
2. Автоматизувати збір релізних артефактів у єдиному сховищі.
3. Запровадити щотижневий dependency report.
4. Підключити `dnsx` і `gau` через відповідні оркестратори з regression-покриттям.

### Дні 61–90

1. Провести контрольний «stress + prelaunch strict + canary» цикл.
2. Виміряти тренд стабільності: кількість fail-гейтів, MTTR, частота rollback.
3. Переглянути policy і зафіксувати версію процесу як стандарт команди.
4. Запустити freemium enrichment lane (`shodan/censys`) у неблокуючому режимі з fallback на baseline.
5. Завершити пілот `DistributedQueueOrchestrator` (Celery+Redis) з rollback на локальний dispatcher.

## 9. Модель виконання для невеликої команди

1. Release owner: відповідає за фінальний verdict і rollout-рішення.
2. Security owner: відповідає за core/tool dependency lanes.
3. Runtime owner: відповідає за prelaunch, артефакти, rollback механізм.
4. Incident owner (черговий): відповідає за triage, ескалацію, пост-мортем.

Правило взаємодії: жодна роль не може одноосібно обійти fail-вердикт гейта без письмового risk acceptance з терміном дії.

## 10. Політики, які потрібно закріпити письмово

1. Політика версіонування залежностей:
   - Core: планові патч-оновлення мінімум раз на 2 тижні.
   - Tool: планові оновлення мінімум раз на місяць.
2. Політика безпеки релізу:
   - Blocker CVE у core = реліз заборонено.
3. Політика артефактів:
   - Реліз без повного комплекту артефактів вважається технічно невалідним.
4. Політика rollback:
   - Rollback має бути перевірений щонайменше раз на квартал.
5. Політика freemium-використання:
   - freemium-модулі не можуть бути обов’язковими для baseline release pass;
   - при quota/rate-limit errors система переходить у degraded enrichment mode без зриву artifact assembly.
6. Політика оркестрації:
   - кожний інструмент підключається тільки через визначений сервіс оркестрації з контрактними тестами.

## 11. Дорожня карта розгортання по середовищах

### 11.1 Локальна підготовка

1. Вирівняти локальні залежності за lock/snapshot.
2. Запустити цільовий набір тестів.
3. Підготувати prelaunch артефакти для перевірки.

### 11.2 CI-інтеграція

1. Запустити lane matrix.
2. Зберегти аудиторські звіти у artifacts.
3. Автоматично згенерувати короткий release-verdict.

### 11.3 Staging

1. Виконати strict prelaunch із повним rehearsal.
2. Перевірити сумісність з опорними сценаріями.
3. Підтвердити готовність rollback.

### 11.4 Production

1. Запустити canary.
2. Відстежити метрики стабільності.
3. Прийняти рішення full rollout/rollback.

## 12. Що вважати завершенням цього плану

План вважається виконаним, коли одночасно виконані умови:

1. Є стабільний релізний цикл без ручних обхідних рішень.
2. Core dependency lane без блокуючих вразливостей.
3. Tool lane має прозорий remediation backlog з SLA.
4. Prelaunch strict стабільно проходить у цільовому профілі.
5. Артефакти релізу валідні зовнішніми інструментами перевірки.
6. Є робочий rollback, перевірений практикою.

## 13. Заключення

Подальше розгортання HANNA має бути не «швидшим за будь-яку ціну», а керованим і доказовим. Найбільша цінність цього плану в тому, що він перетворює невизначеність на послідовність формальних кроків із вимірюваним результатом. Він не залежить від конкретної людини, не спирається на імпровізацію і зменшує ймовірність повторення старих інцидентів.

Критичний меседж простий: стабільність не з’являється сама собою. Вона виникає з дисципліни: чітких меж, чітких гейтів, чітких ролей, чітких критеріїв зупинки. Якщо дотримуватися цього майстер-плану без розширення scope і без «разових винятків», система отримає передбачуваний шлях до безпечного масштабування, а команда — контроль над ризиком і вартістю експлуатації.

## 14. Розширений контур аргументів для подальшого розгортання

### 14.1 Чому саме послідовний rollout, а не «великий стрибок»

Послідовний rollout з контрольними точками критично важливий для систем, де є асинхронні воркери, зовнішні модулі і багатоформатні артефакти. Великий стрибок одразу на повний обсяг навантаження ускладнює аналіз: коли виникає збій, важко відрізнити, це проблема коду, конфігурації, середовища чи залежності. Canary-підхід вирішує це за рахунок раннього сигналу і обмеження blast radius.

Аргумент практичний: навіть якщо дефект не критичний, його вартість на повному rollout у кілька разів більша за вартість локалізації на canary. Тому послідовність «strict gate -> canary -> full rollout» має бути непорушною політикою, а не рекомендацією.

### 14.2 Чому потрібно тримати окремий журнал рішень

У проектах з високою частотою змін головною проблемою стає втрата контексту: команда пам’ятає, що щось «вже виправляли», але не пам’ятає чому, як і за яких обмежень. Журнал рішень (decision log) знімає це навантаження: кожна нетривіальна зміна фіксується з коротким обґрунтуванням, альтернативами і критерієм перегляду.

Це прямо впливає на розгортання: коли в проді стається аномалія, on-call бачить не лише поточний стан, а й логіку попередніх компромісів. В результаті зменшується кількість помилкових rollback або небезпечних «швидких» патчів.

### 14.3 Економіка підтримки і технічний борг

Технічний борг у такій системі дорожчає нелінійно. На ранньому етапі він виглядає як кілька незручних скриптів або неідеальна структура залежностей. На етапі масштабування той самий борг перетворюється на регулярні збої, втрату часу на triage і падіння швидкості релізів. Тому стабілізаційні інвестиції мають найвищий ROI саме до масштабного розширення функціоналу.

Окремо: борг у dependency governance найпідступніший, бо проявляється не одразу. Сьогодні це попередження в аудиторському звіті, завтра — блокуючий інцидент на релізі. Саме тому в плані закладена регулярна, а не разова, практика аудиту.

### 14.4 Політика комунікації під час релізу

Проблема багатьох команд — відсутність стандарту комунікації в «гарячу» фазу rollout. Для HANNA потрібно закріпити простий формат:

1. Що змінюємо.
2. Який очікуваний ефект.
3. Які сигнали вважаються нормою.
4. Які сигнали є умовою негайної зупинки.
5. Хто приймає фінальне рішення про продовження або rollback.

Це прибирає двозначність і знижує ризик колективної помилки, коли ніхто не хоче «натиснути стоп» першим.

### 14.5 Довгострокова стабільність через рутину

Стабільність не підтримується героїчними ривками. Вона підтримується рутиною: регулярним аудитом залежностей, контрольованими оновленнями, перевіреним prelaunch, короткими пост-релізними звітами і дисципліною rollback-перевірок. Якщо ці ритуали виконуються послідовно, система входить у стан передбачуваного операційного циклу: менше сюрпризів, швидше відновлення, вища довіра до релізів.

Фінальний аргумент: майстер-план має сенс лише тоді, коли він зменшує кількість «ручних героїчних дій». Якщо після впровадження плану команда продовжує рятувати релізі вручну, план треба переглядати. Якщо ж більшість рішень приймається заздалегідь визначеними правилами, план працює правильно.
